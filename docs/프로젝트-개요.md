# 부산MBC 온에어 — 프로젝트 개요 (처음 보는 사람용)

방송국(부산MBC)의 TV/라디오를, **로그인한 시청자가 휴대폰·PC 웹으로 실시간 시청**하는 앱입니다.
지금 실제로 인터넷에 배포되어 동작 중입니다: **https://mybusanmbc.duckdns.org**

이 문서 하나로 "무엇을 / 어떻게 / 얼마에" 만들었는지 알 수 있게 정리했습니다.

---

## 1. 한눈에 보기

- **무엇**: 방송 실시간 스트리밍 웹앱 (로그인 → 채널 선택 → 재생).
- **누구**: 방송국 내부/지정 시청자(로그인 필요). 관리자(방송국 직원)는 채널을 추가/수정.
- **형태**: 웹앱이면서 **PWA** — 아이폰/안드로이드 홈 화면에 아이콘으로 설치해 "앱처럼" 사용.
- **비용**: 현재 **월 $0**(무료 인프라만 사용). 자세한 근거는 7장.
- **철학**: 빌드 도구 없이 순수 HTML/CSS/JS + 무료 오픈소스·무료 클라우드로 구성. 단계(Phase)별로 "동작 확인 → 다음 단계".

---

## 2. 시청자가 쓰는 흐름

```
① 브라우저(또는 홈 화면 앱)로 https://mybusanmbc.duckdns.org 접속
② 로그인 (이메일/비밀번호)
③ 채널 목록에서 원하는 채널(예: TV1) 선택
④ 영상 재생 (라이브, 약 10~20초 지연은 정상)
```

방송국 측(송출)은 별도로 **OBS**라는 프로그램으로 서버에 영상을 "쏘아 보냅니다".

---

## 3. 전체 구조 (아키텍처)

```
[방송용 PC]                     [Oracle 클라우드 서버 (24시간 켜짐)]                [시청자]
 OBS Studio                       ┌─────────────────────────────────────┐         브라우저 / 홈앱
   │                              │  MediaMTX  (미디어 서버)             │           │
   │ RTMP 송출(인증)              │   :1935 RTMP 수신 → :8888 HLS 변환   │           │
   └──────────────────────────▶  │                                     │           │
   rtmp://도메인/live/tv1         │  Caddy  (웹 서버 + 리버스 프록시)    │ ◀─HTTPS──┘
   (user/pass 필요)              │   :443 HTTPS(자동 인증서)            │  https://mybusanmbc.duckdns.org
                                  │    /        → 정적 프론트(HTML/JS)   │
                                  │    /live/*  → MediaMTX(:8888)로 전달 │
                                  └─────────────────────────────────────┘
                                              ▲
                                              │ 로그인·회원·채널목록은
                                              ▼
                                     [Supabase 클라우드] (인증 + 데이터베이스)
```

핵심 아이디어 3가지:
1. **송출(OBS) → 미디어서버(MediaMTX) → 시청(브라우저)** 로 흐르는 실시간 파이프라인.
2. **같은 주소(same-origin)** 로 웹과 영상을 모두 제공(아래 4-3에서 이유 설명).
3. **로그인/데이터는 Supabase(클라우드)** 가 대신 처리 → 서버에 DB를 직접 안 깔아도 됨.

## 4. 기술 구현 상세 (구성요소별)

### 4-1. 방송 송출 — OBS Studio
- 방송용 PC에서 **OBS**(무료)로 화면/영상을 **RTMP** 프로토콜로 서버에 전송.
- 전송 주소: `rtmp://mybusanmbc.duckdns.org/live` + 스트림 키 `tv1?user=publisher&pass=<비밀번호>`.
- **인증**: 스트림 키에 계정/비번을 붙여야만 송출 허용 → 아무나 방송을 밀어넣지 못함.
- OBS 권장 설정(실측으로 확정): **키프레임 간격 2초**(A/V 싱크), **캔버스=출력 해상도 같은 16:9**(검은 띠 방지).

### 4-2. 미디어 서버 — MediaMTX (핵심)
- **역할**: OBS가 보낸 RTMP 영상을 받아, 웹 브라우저가 재생 가능한 **HLS** 형식으로 실시간 변환.
  - RTMP(:1935)로 수신 → HLS(:8888)로 출력. HLS는 영상을 짧은 조각(.ts/.m3u8)으로 잘라 HTTP로 전달하는 방식이라 웹 표준 재생에 적합.
- **왜 MediaMTX**: 단일 실행파일, 가볍고(트랜스코딩 아니라 단순 리먹싱이라 CPU/RAM 거의 안 씀), RTMP·HLS·인증을 한 번에 지원. 오픈소스(무료).
- **인증 설정(`authInternalUsers`)**: 익명은 **읽기(재생)만**, 지정 계정(`publisher`)만 **송출** 허용.
- **상시 변환(`hlsAlwaysRemux`)**: 송출 중이면 시청자가 없어도 HLS를 계속 생성 → "온에어" 상태 확인과 재생이 지연 없이 즉시 동작.
- **HLS 방식 `fmp4`**: 프로덕션 검증 결과 fmp4(CMAF)로 운영(싱크 안정).
- 프로세스 관리: **systemd 서비스**로 등록 → 서버 부팅 시 자동 시작, 죽으면 자동 재시작.

### 4-3. 웹 서버 + 리버스 프록시 — Caddy (핵심)
- **역할 1 (정적 서빙)**: 프론트엔드 HTML/CSS/JS 파일을 웹으로 제공.
- **역할 2 (리버스 프록시)**: `/live/*` 요청만 내부 MediaMTX(:8888)로 전달. 나머지는 정적 파일.
- **역할 3 (자동 HTTPS)**: 도메인만 지정하면 Let's Encrypt에서 **SSL 인증서를 자동 발급·갱신** → 자물쇠(🔒) 붙은 https 제공. 별도 인증서 작업 불필요.
- **왜 same-origin(같은 출처)로 묶었나 — 중요한 이유**:
  - 페이지(웹)와 영상 스트림을 **다른 주소(포트)** 로 두면, 특히 **iOS 사파리**에서 교차 출처(CORS)·쿠키 문제로 재생이 실패했음(실제 겪은 버그).
  - Caddy가 웹과 `/live` 영상을 **하나의 도메인·포트(443)** 로 함께 제공 → 브라우저 입장에선 전부 같은 출처 → 문제 사라짐.
  - 프론트 코드는 재생 주소를 "현재 접속한 주소(location) 기준"으로 만들기 때문에 **도메인이 바뀌어도 코드 수정이 필요 없음**.

### 4-4. 프론트엔드 — 순수 HTML/CSS/JS (빌드 없음)
- React 같은 프레임워크·빌드 도구 없이 **바닐라 웹 기술**만 사용 → 단순·가볍고 유지보수 쉬움.
- **영상 재생**: `hls.js` 라이브러리 사용. 단, **아이폰 사파리는 브라우저 자체가 HLS 재생을 지원**하므로 그쪽은 네이티브 재생으로 분기.
  - (주의점) 크롬은 "HLS 재생 가능?" 질문에 애매하게 "maybe"라고 답하지만 실제론 못 하므로, **hls.js를 우선 사용**하도록 순서를 잡음(실제 겪은 버그의 해결책).
- **온에어/오프에어 표시**: 각 채널의 HLS 주소를 가볍게 조회해 재생 가능 여부로 상태 뱃지 표시.
- 순수 함수(입력검증·URL조립·폼검증)는 **Node 기반 단위 테스트**로 검증.

### 4-5. 로그인 · 데이터베이스 — Supabase
- **역할**: 회원가입/로그인(이메일·비밀번호) + 채널 정보 저장(PostgreSQL 데이터베이스).
- **왜 Supabase**: 서버에 DB·인증 서버를 직접 구축하지 않고, 클라우드가 대신 처리(무료 플랜). 프론트에서 공개용 키로 직접 호출.
- **보안(RLS, Row-Level Security)**: 데이터베이스 차원에서 "관리자만 채널을 수정 가능" 같은 권한을 강제. 프론트를 우회해도 규칙이 지켜짐.
- 관리자 지정: `profiles.is_admin = true` 인 계정만 관리 화면(채널 CRUD) 사용.

### 4-6. 앱처럼 만들기 — PWA
- **manifest.json**(앱 이름·아이콘·전체화면 설정) + **service worker**(앱 화면 파일 캐시) + **아이콘**(마스코트 기반).
- 아이폰: Safari → "홈 화면에 추가" → 아이콘 실행 시 **주소창 없는 전체화면 앱**.
- **중요 설계**: 서비스워커는 **앱 껍데기(HTML/CSS/JS)만 캐시**하고, **영상·로그인·설정은 절대 캐시 안 함**(라이브 데이터라 캐시하면 오류). 코드가 바뀌면 캐시 버전을 올려 새로 반영.
- (참고) 네이티브 앱(앱스토어 등록)은 Mac·개발자 비용이 필요해 이번 범위 밖. 지금 PWA는 나중에 네이티브로 감쌀 때 그대로 재활용 가능.

### 4-7. 디자인
- **MBC 브랜드**: 메인 컬러 퍼플(#6706D3), MBC NEW 폰트, 파란 마스코트 캐릭터, 라이트(밝은) 테마. 온에어는 레드.
- claude.ai/design으로 고도화한 디자인을 가져와 반영(그라데이션 로그인·카드형 채널·라운드 플레이어). **기능 로직은 건드리지 않고 화면만** 개선.

## 5. 배포(서버 구축) 상세

### 5-1. 클라우드 서버 — Oracle Cloud
- **Oracle Cloud Always Free**(평생 무료) 등급으로 가상 서버(인스턴스) 1대 생성.
- 사양: **VM.Standard.E2.1.Micro**(AMD, 1 OCPU / 1GB RAM), **Ubuntu 22.04**, 리전 도쿄.
  - (참고) 원래 ARM(고사양 무료)을 노렸으나 물량 부족으로 실패 → AMD 무료 사양으로 진행. 우리 워크로드엔 충분(트랜스코딩이 아니라 단순 변환이라 가벼움).
- **고정 공인 IP**(예약): `158.101.159.134` — 재부팅해도 안 바뀌게 예약 IP 부여(도메인 연결 안정).

### 5-2. 도메인 + HTTPS — DuckDNS + Caddy
- **DuckDNS**(무료): `mybusanmbc.duckdns.org` 서브도메인을 서버 IP에 연결.
- **Caddy**가 그 도메인으로 **무료 SSL 인증서 자동 발급** → https 접속.

### 5-3. 방화벽 — 2겹 개방
- Oracle은 방화벽이 **두 겹**이라 둘 다 열어야 외부 접속 가능:
  1. **OCI 보안 목록**(클라우드 콘솔) — 포트 80/443/1935 인그레스 허용.
  2. **Ubuntu 내부 방화벽(iptables)** — 같은 포트 허용.
- (실제 겪은 이슈) iptables에서 "전부 거부(REJECT)" 규칙이 허용 규칙보다 위에 있어 포트가 막혔고, 이 때문에 HTTPS 인증서 발급이 실패 → REJECT를 맨 뒤로 옮겨 해결.

### 5-4. 앱 배포 + 서비스 등록
- 서버에 코드를 GitHub에서 clone(`/opt/mbc-app`), MediaMTX 바이너리 설치(`/opt/mediamtx`).
- 비밀값(`config.js`의 Supabase 키, MediaMTX 송출 비번, 인증서 이메일)은 **서버에만** 두고 GitHub엔 올리지 않음.
- MediaMTX는 systemd, Caddy는 apt 설치 시 자동 서비스 등록 → **부팅 시 자동 시작**.
- Supabase 대시보드에 프로덕션 도메인 등록(로그인 이메일 흐름용).

### 5-5. 업데이트(코드 반영) 방법
```bash
# 서버에 SSH 접속 후
cd /opt/mbc-app && sudo git pull
sudo chmod -R a+rX /opt/mbc-app/frontend
# 프론트 코드가 바뀌었으면 service-worker.js의 캐시 버전(vN)을 올려 재방문자에게 반영
```

## 7. 비용 — 정확히 확인 (현재 월 $0)

결론: **현재 지불하는 비용은 없습니다.** 단, 서비스별로 "무료의 조건"과 주의점을 정확히 정리합니다.

### ✅ 확실히 무료 (조건 없음)
| 항목 | 근거 |
|------|------|
| **Caddy** | 오픈소스(무료 SW). HTTPS 인증서(Let's Encrypt)도 무료. **비용 개념 자체가 없음.** |
| **MediaMTX** | 오픈소스(무료 SW). 비용 없음. |
| **DuckDNS** | 무료 서브도메인 서비스. 비용 없음. |
| **OBS Studio** | 무료 오픈소스 방송 프로그램. 비용 없음. |

### ✅ 무료지만 "조건/한도"가 있는 것
**Oracle Cloud (Always Free)** — 우리가 쓰는 자원은 평생 무료 등급 안:
- 컴퓨트 **E2.1.Micro**(무료 대상 사양) ✓
- 부팅 디스크 ~47GB(무료 한도 200GB 내) ✓
- **아웃바운드 트래픽 월 10TB**(무료) — 우리 사용량은 극히 일부 ✓
- ⚠️ **공인 IPv4 주소**: 실행 중인 무료 인스턴스에 **붙어 있는 동안**은 무료 사용 범위입니다. 다만 오라클이 최근 공인 IPv4 정책을 바꾼 바 있어, **미사용(분리된) 예약 IP는 소액 과금될 수 있음** → **IP는 인스턴스에 계속 붙여두면 됩니다.**
- ⚠️ **유휴 회수**: Always Free 인스턴스가 장기간 거의 idle이면 오라클이 회수할 수 있음(과금은 아님). 실제 사용 중이면 문제 없음.
- 👉 **권장 확인**: 오라클 콘솔 → **Billing & Cost Management → Cost Analysis** 에서 **월 청구액 $0** 인지 한 번 확인. 결제수단은 본인확인용이며, Always Free 자원만 쓰면 청구되지 않습니다.

**Supabase (Free plan)** — 소규모 앱엔 충분:
- 무료 한도: DB 500MB, 파일 1GB, 월 활성 사용자 5만 명, 대역폭 등 — 이 앱(로그인 + 채널 테이블)엔 여유롭게 무료 ✓
- ⚠️ **7일 무활동 시 프로젝트 일시정지**: 일주일 동안 아무도 로그인/사용 안 하면 프로젝트가 pause됨(대시보드에서 재개 클릭 필요). **평상시 사용 중이면 계속 활성**입니다.

## 8. 알아두면 좋은 운영 주의점

- **송출은 채널당 한 곳만**: 같은 스트림 키(tv1)로 두 PC가 동시에 송출하면 충돌. 실제 방송 PC로 넘어갈 땐 테스트 송출을 먼저 중지.
- **재생 지연**: HLS 특성상 실시간 대비 10~20초 지연은 정상.
- **디자인/코드 변경 반영**: 서버 `git pull` 후, 프론트가 바뀌었으면 서비스워커 캐시 버전을 올려야 재방문자에게 즉시 반영(안 그러면 새로고침 후 반영).
- **비밀값 관리**: Supabase 키·송출 비번·인증서 이메일은 서버에만. GitHub 저장소엔 올리지 않음(자리표시자만).
