# 방송 스트림 모니터링 앱 — 구현 계획서

> 방송국 TV/라디오 소스를 일반 시청자가 휴대폰으로 시청하는 공개형 웹앱.
> 로그인 기반, 저지연 불필요, 소규모 동시접속, 무료 인프라 우선.

---

## 0. 프로젝트 개요

| 항목 | 결정 |
|------|------|
| 서비스 형태 | 공개형 (로그인/회원가입 있음) |
| 실시간성 | 저지연 불필요 → **일반 HLS** 사용 (10~20초 지연 허용) |
| 동시접속 | 소규모 (초기 무료 인프라로 충분) |
| 개발 리소스 | 1인 개발 |
| 월 고정비 목표 | 도메인 비용(연 1~2만원) 외 사실상 $0 |

### 확정 기술 스택
- **미디어 서버**: MediaMTX (오픈소스, 무료) — RTMP/SRT 수신 → HLS 변환
- **서버 인프라**: Oracle Cloud Always Free (ARM, 월 10TB 트래픽 무료)
- **로그인/DB**: Supabase 무료 티어 (Auth + Postgres)
- **리버스 프록시/SSL**: Caddy (Let's Encrypt 자동)
- **프론트엔드**: 모바일 웹앱(PWA) + hls.js
- **도메인**: 저렴한 도메인 또는 DuckDNS(무료)

### 전체 아키텍처
```
방송 인코더 ──RTMP/SRT──▶ MediaMTX(HLS 변환) ──▶ Caddy(HTTPS) ──▶ 웹앱(hls.js)
   (원본)                   [Oracle Cloud 서버]                        │
                                                                  Supabase Auth
                                                                (로그인/사용자DB)
```

---

## ⚠️ 사전 확인사항 (개발 시작 전 필수)

방송 소스를 **일반 대중에게 공개**하면 저작권·재송신 권리 문제가 발생할 수 있음.
직원 내부 모니터링과 법적 성격이 다르므로, 개발 착수 전 **편성·법무 부서와 공개 시청 권리 범위를 반드시 확인**할 것. 이는 기술이 아니라 서비스 존폐를 가르는 문제.

아래 계획은 권리 정리가 완료되었다는 전제로 작성됨.

---

## 개발 원칙

- **한 번에 다 만들지 않는다.** 각 Phase는 "동작하는 상태"를 확인하고 다음으로 넘어간다.
- **로컬에서 전부 검증한 뒤 서버에 올린다.** (Phase 1~4 로컬 → Phase 5 배포)
- **인증/접근제한은 나중에.** 초기엔 "로그인해야 페이지가 보인다" 수준으로 시작, 서명 URL은 Phase 6에서.

---

## Phase 1 — 스트림을 흐르게 하기 (로컬)

> 목표: 영상 소스가 HLS로 재생되는 것을 증명한다. 이게 되면 절반은 온 것.

- [ ] MediaMTX 로컬 설치 (바이너리 다운로드 또는 Docker)
- [ ] `mediamtx.yml` 설정 파일 작성 — RTMP 수신 + HLS 출력 활성화
- [ ] 테스트 소스 준비 (둘 중 택1)
  - [ ] 실제 방송 인코더에서 RTMP push
  - [ ] FFmpeg로 테스트 영상 push (`ffmpeg -re -stream_loop -1 -i sample.mp4 -c copy -f flv rtmp://localhost/live/test`)
- [ ] 라디오(오디오 전용) 스트림도 동일하게 테스트
- [ ] 생성된 HLS 주소(`http://localhost:8888/test/index.m3u8`)를 VLC 또는 브라우저로 재생 확인

**완료 기준**: VLC에서 스트림이 재생된다.

---

## Phase 2 — 웹 플레이어 (로컬, 로그인 없음)

> 목표: 브라우저에서 HLS를 재생하는 단순 페이지.

- [ ] HTML 페이지 + hls.js 연동
- [ ] iOS 사파리는 HLS 네이티브 지원 / 안드로이드·크롬은 hls.js 사용 → 분기 처리
- [ ] 채널 목록 UI (TV 채널들 + 라디오)
- [ ] 채널 클릭 시 해당 HLS 재생
- [ ] 모바일 화면 레이아웃 확인 (반응형)

**완료 기준**: 폰 브라우저에서 채널을 골라 재생할 수 있다.

### 예상 파일 구조 (프론트)
```
/frontend
  index.html          # 로그인 화면 (Phase 3에서 추가)
  player.html         # 채널 목록 + 플레이어
  /js
    player.js         # hls.js 초기화 및 재생 로직
    channels.js       # 채널 목록 데이터
    auth.js           # Supabase 인증 (Phase 3)
  /css
    style.css
  manifest.json       # PWA (Phase 7)
  service-worker.js   # PWA (Phase 7)
```

## Phase 3 — Supabase 로그인 얹기

> 목표: 로그인한 사용자만 플레이어 페이지에 접근.

- [ ] Supabase 프로젝트 생성 (무료 티어)
- [ ] Auth 설정: 이메일/비밀번호 로그인 활성화 (필요 시 소셜 로그인 추가)
- [ ] 회원가입 / 로그인 / 로그아웃 화면 구현
- [ ] 로그인 세션 확인 → 미로그인 시 플레이어 페이지 접근 차단, 로그인 화면으로 리다이렉트
- [ ] 사용자 테이블 설계 (필요 시: 권한 등급, 시청 가능 채널 등)
- [ ] 비밀번호 재설정 흐름 (Supabase 기본 제공)

**완료 기준**: 로그인해야만 채널을 볼 수 있다.

---

## Phase 4 — 채널 관리 및 UI 다듬기 (로컬)

> 목표: 실사용 가능한 수준의 완성도.

- [ ] 채널 목록을 Supabase DB에서 불러오기 (하드코딩 → DB화)
- [ ] 채널 썸네일/이름/상태(온에어 여부) 표시
- [ ] 재생 오류 처리 (스트림 끊김 시 재접속, 에러 메시지)
- [ ] 로딩 상태 표시
- [ ] 라디오 채널은 오디오 전용 UI로 분기

**완료 기준**: 로컬에서 서비스로 써도 될 만한 UX가 나온다.

---

## Phase 5 — 공개 배포 (Oracle Cloud)

> 목표: 로컬에서 검증한 것을 실제 공개 서버로 이전.

- [ ] Oracle Cloud 계정 생성 및 Always Free ARM 인스턴스 생성
- [ ] 방화벽/보안목록에서 필요한 포트 개방 (80, 443, RTMP 등)
- [ ] 서버에 MediaMTX 설치 및 `mediamtx.yml` 이전
- [ ] 방송 인코더 → Oracle 서버로 RTMP/SRT push 경로 구성 (인제스트 보안 주의)
- [ ] 도메인 준비 (구매 또는 DuckDNS) 및 DNS A레코드 연결
- [ ] Caddy 설치 → 자동 HTTPS(SSL) 발급 → HLS/웹앱 리버스 프록시
- [ ] 프론트엔드 배포 (Caddy로 정적 서빙 또는 별도 호스팅)
- [ ] 외부(모바일 데이터)에서 접속 테스트

**완료 기준**: 폰에서 도메인으로 접속해 로그인 후 시청된다.

### 참고: 인제스트 보안
- RTMP 수신 포트를 인터넷에 그냥 열면 누구나 방송을 밀어넣을 수 있음.
- MediaMTX의 publish 인증(사용자/비밀번호) 또는 방화벽 IP 제한을 반드시 걸 것.

## Phase 6 — 접근 제한 (서명 URL)

> 목표: 로그인 안 한 사람이 HLS 주소를 직접 알아내도 재생 불가하게. (난이도 있음 → 안정화 후 진행)

- [ ] Supabase Edge Function으로 서명 URL 발급 엔드포인트 작성
- [ ] 로그인한 사용자 요청 시 → 만료시간 포함 토큰이 붙은 임시 HLS URL 반환
- [ ] MediaMTX 또는 Caddy에서 토큰 검증 후에만 스트림 서빙
- [ ] 토큰 만료(예: 수 분) 및 갱신 로직

**완료 기준**: 스트림 주소가 유출돼도 곧 만료되어 무단 시청이 불가하다.

---

## Phase 7 — PWA로 앱처럼 만들기

> 목표: 폰 홈화면에 설치되는 앱 경험. (부담 적음)

- [ ] `manifest.json` 작성 (앱 이름, 아이콘, 시작 화면)
- [ ] Service Worker 등록 (오프라인 셸, 설치 유도)
- [ ] iOS/안드로이드에서 "홈 화면에 추가" 동작 확인
- [ ] 앱 아이콘 및 스플래시 이미지

**완료 기준**: 폰 홈에서 앱 아이콘으로 실행된다.

---

## Phase 8 — 운영/모니터링 (선택)

- [ ] 스트림 상태 헬스체크 (채널 다운 감지 및 알림)
- [ ] Oracle 트래픽 사용량 모니터링 (10TB 한도 관리)
- [ ] 동시접속자/시청시간 로깅
- [ ] 시청자 증가 시 확장 전략 검토 (엣지 서버 추가 등)

---

## 대역폭 참고 (중요)

소프트웨어는 전부 무료지만 **업로드 대역폭**은 유일한 실제 제약임.
- 예: 2Mbps 스트림 × 동시 20명 = 40Mbps 지속 업로드
- Oracle 무료 10TB/월 = 2Mbps 기준 약 수천 시청시간 커버 → 초기엔 충분
- 시청자가 급증하면 그때 엣지 서버 추가를 검토
